# 新底盘硬件接入指南

当前分支已经从旧底盘运行路径脱钩，目标是完整支持 STM32 双 UART 4WD 新底盘和 Hikrobot GigE 相机。默认 profile 是 `stm32_hoverboard_4wd`，不再维护 `campus_car` 旧底盘 profile。

## 目录

```text
config/
  robot.env                         # 通用默认值和 profile loader，默认 stm32_hoverboard_4wd
  robot.local.env                   # 本机覆盖，Git 忽略
  profiles/
    stm32_hoverboard_4wd.env        # 新底盘默认 profile
    template.env                    # 新 STM32/Hikrobot profile 模板
hardware/
  hoverboard_driver/                # STM32 双 UART ros2_control 驱动
scripts/
  stm32_hoverboard_probe.sh         # 依赖和串口探测
  stm32_hoverboard_start.sh         # 启动底盘驱动
  hikrobot_camera_probe.sh          # 相机探测
  hikrobot_camera_start.sh          # 启动 camera_aravis2
```

## 常用命令

```bash
./scripts/deploy_dependencies.sh
./scripts/stm32_hoverboard_probe.sh
./scripts/hikrobot_camera_probe.sh
./scripts/launch_all.sh
./scripts/check_all.sh
./scripts/stop_all.sh
```

Docker 隔离入口：

```bash
./scripts/docker_build.sh
./scripts/docker_run_stm32.sh
```

Docker 只是隔离 ROS2/依赖和 profile 入口；宿主机仍负责 USB/TTL 串口、网卡、udev 和急停。详细见 `docs/Docker部署指南.md`。

## STM32 双串口 4WD 底盘

卖家确认新底盘使用 hoverboard-driver 风格串口协议：

```text
start = 0xABCD
int16 steer
int16 speed
uint16 checksum = start ^ steer ^ speed
```

串口为 `115200 8N1`、`5V TTL`、`RX/TX/GND`。实车有前后两个驱动器，每个驱动器控制两个电机，因此需要两个串口。

默认 profile：

```bash
config/profiles/stm32_hoverboard_4wd.env
```

本地需要按实际插线修改串口路径，建议使用稳定的 `/dev/serial/by-id/...`：

```bash
cat > config/profiles/stm32_hoverboard_4wd.local.env <<'EOF'
HOVERBOARD_FRONT_DEVICE="/dev/serial/by-id/<front-driver>"
HOVERBOARD_REAR_DEVICE="/dev/serial/by-id/<rear-driver>"
EOF
```

该 profile 默认通过 `scripts/stm32_hoverboard_start.sh` 启动项目内 `hardware/hoverboard_driver`。首次部署会构建到 `hoverboard_ws/install`，构建产物不提交到 Git。

卖家确认 `steer` 和 `speed` 的协议范围是 `[-1000, 1000]` RPM。工程默认保守处理：profile 里 `HOVERBOARD_COMMAND_LIMIT_RPM=50`，驱动会在串口写出前对 `steer/speed` 做限幅；同时 `DEFAULT_LINEAR_SPEED=0.10`、`MAX_LINEAR_SPEED=0.30`、`DEFAULT_ANGULAR_SPEED=0.20`、`MAX_ANGULAR_SPEED=0.60`。首次实车测试必须先架空轮子或断开电机负载。

当前实际驱动器没有编码器 tick 数据，所以项目内的 `hoverboard_driver` 默认使用卖家回复的 10ms 短反馈帧，并把 `diff_drive_controller` 配成 `open_loop: true`、`position_feedback: false`。短反馈帧可用字段包括左右轮实测 RPM、电压、温度和 LED 字段；没有 `wheelR_cnt/wheelL_cnt`，因此不能当作真实编码器里程计。后续如果换成完整 hoverboard 固件反馈，可把 `HOVERBOARD_FEEDBACK_FORMAT=full` 后再重新测试。

卖家没有给出明确的串口超时停车时间；急停目前是通过通断 TTL 通信控制线实现。软件侧在驱动关闭时会先写一次零速命令，但实车仍应保留独立硬件急停/断线方案。

## Hikrobot GigE 相机

已选相机：`MV-CS016-10GC`。本工程优先用 `camera_aravis2` 接入，因为该相机兼容 GigE Vision / GenICam。

到货后先探测：

```bash
./scripts/hikrobot_camera_probe.sh
```

如果能看到相机 GUID 或 IP，写入本地覆盖：

```bash
cat >> config/profiles/stm32_hoverboard_4wd.local.env <<'EOF'
HIKROBOT_CAMERA_GUID="<GUID或IP>"
EOF
```

如果 `arv-tool-0.8` 或 `camera_finder` 看不到相机，优先检查 NUC 网卡与相机是否同网段、PoE/外部供电是否正常、防火墙是否阻断 GigE Vision。若 `camera.log` 报 `PixelFormat` 不支持，先把 `HIKROBOT_PIXEL_FORMAT` 从 `BGR8` 改成 `RGB8`。

## 上层接口

UE、RTK、视频和控制层继续使用稳定接口：

```text
tcp://<NUC_IP>:9090
http://<NUC_IP>:8888/robot_cam/index.m3u8
http://<NUC_IP>:8080/
/U2RTopic_Command
/R2UTopic_Pos
/R2UTopic_Text
/cmd_vel
/fix
/odom
```

新底盘仍通过 `/cmd_vel` 接收 `geometry_msgs/Twist`，由 `hoverboard_driver` 和 `diff_drive_controller` 转成前后两块 STM32 驱动器的串口命令。
